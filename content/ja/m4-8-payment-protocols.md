# 4.8  エージェント決済プロトコルに対応する

## 概要と理由
エージェントが HTTP 呼び出しの瞬間に決済を清算しなければならないとき - メーター制の API アクセスを購入する、計算資源に支払う、注文をチェックアウトする - それは、人間がカード番号を打ち込むことなく行わなければなりません。エージェントネイティブな決済は、HTTP の `402 Payment Required` ハンドシェイクに応答する **清算 (settlement)** 層（x402、MPP）と、ユーザーが本当に購入を承認したことを証明する **認可 (authorization)** 層（AP2）に分かれます。

- **x402**（Coinbase）は、最小限の暗号資産ネイティブのケースです。1リクエスト、1回のステーブルコイン決済（USDC、たいてい Base 上）、1レスポンス。これは機械対機械の従量課金 - API 呼び出し、データフィード、計算資源、エージェント間サービス - のために専用設計されており、**ステーブルコイン専用** なので、カード/銀行レールや物理商品のチェックアウトには届きません。
- **MPP**（Machine Payments Protocol、Stripe と Tempo 発、現在は IETF Internet-Draft）は、同じ `402` の交換を **決済手段非依存** のフレームワークに一般化します。単一のエンドポイントが、ステーブルコイン、カード、銀行振込、さらには Bitcoin まで受け付けられ、2つの intent を担います。`charge`（一回限り - これは x402 の決済に直接対応し、MPP を x402 と後方互換にします）と `session`（支出上限を一度事前認可し、その後きめ細かいマイクロペイメントをストリームする）です。その幅広さゆえに、MPP は呼び出しごとの従量課金に **限定されません** - 同じハンドシェイクが、暗号資産 *と* 従来のレールの両方にまたがって、**物理商品とサービス** を清算します。
- **AP2**（Agent Payments Protocol、Google）は、ひとつ上の層に位置します。`402` を清算する代わりに、**ユーザーがこの購入を承認したという暗号学的な証明** - 署名された *Mandate*（何が認可されたかについての SD-JWT の *Checkout Mandate*、open/事前認可 または closed/最終 の形で、加えて資金手段についての *Payment Mandate*）- を運びます。A2A の拡張として作られ、あなたの `agent-card.json` で広告され、決済手段非依存で、清算層を置き換えるのではなく **組み合わせます**。AP2 が *認可* し、x402 と MPP が *清算* します。

これらは初期段階の標準です - 採用は急速に伸びていますが、仕様はまだ動いています - なので、これは必須要件ではなく、先行ポジショニングとして扱ってください。

## スコアリング
- **工数 4/5** - 価格の付いたすべてのルート上の決済ミドルウェア、資金の入ったウォレット、ファシリテーターの配線、清算の照合 - 加えて、ウォレットを保有・保護することの恒常的な運用負荷。プロトコルはまだ流動的なので、その上に仕様の変動を見込んでください。
- **インパクト 2/5** - 先を見据えたものです。メーター制アクセスを販売する、またはエージェントネイティブなチェックアウトを今日受け付けるサイトにとっては現実的ですが、採用は初期段階で、待つことの下振れは小さいです。
- **視覚的変化: なし** - `/.well-known/*` のディスカバリーファイルと、有料ルート上の HTTP `402` レスポンス。人間に見えるページは変わりません。

## 手順
1. **暗号資産ネイティブのリクエスト課金には x402。** x402 は、HTTP 越しにインラインで決済をネゴシエートし、暗号資産レール上でステーブルコインで清算します。エージェントが価格の付いたルートを呼び出すと、サーバーは `402 Payment Required` を、決済条件 - 金額、受け入れるスキーム、支払い先 - とともに返します。エージェントはその条件を清算し - 通常は転送を仲介・検証する *ファシリテーター* を通じて - 支払いの証明とともにリトライすると、本物のレスポンスが返ります。対応するには、有料ルートに印を付け、それらをファシリテーターと受け取りウォレットに向け、どのルートが有料かを広告して、エージェントが呼び出す前に有料サーフェスを見つけられるようにします。一般的なウェブフレームワーク向けのオープンソースミドルウェアが、配線をほぼ設定だけにします。
2. **マルチレールと物理商品の清算には MPP。** MPP は `402` に `WWW-Authenticate: Payment` チャレンジで応答し、その `method`（`tempo`、`stripe`、`card`、`lightning`、…）と `intent`（一回限りの `charge`、ストリーミングの `session`）がレールと決済の形を選びます - そのため、1つの連携でステーブルコイン、カード、法定通貨をカバーし、メーター制の呼び出しだけでなく物理商品のチェックアウトに届きます。`/openapi.json`（[4.1](./m4-1-openapi-spec.md)）で支払い可能なオペレーションに注釈を付けてエージェントが有料サーフェスを発見できるようにし、MPP ミドルウェア（SDK は TypeScript、Python、Rust 向けに提供）を配線してハンドシェイクと清算を処理します。
3. **認可の証明には AP2。** エージェントがユーザーに代わって取引する場合、`/.well-known/agent-card.json` 上で AP2 サポートを A2A 拡張として広告し（拡張 URI `https://github.com/google-agentic-commerce/ap2/v1`）、署名された Mandate をチェックアウト経路で受け入れます。これが、その場にいない買い手が *この* カートを *この* 価格で本当に認可したことを、マーチャントやカードネットワークが信頼できるようにするもの - x402/MPP の清算が乗る信頼層です。
4. **認証とエラーのスタックを再利用する。** すべての決済ルートは [3.1](./m3-1-oauth-discovery.md) の OAuth スコープの背後に置かれ、[4.2](./m4-2-rate-limits-and-errors.md) の構造化されたエラーエンベロープを返します。拒否された決済は、正直な `retry_hint` を伴う `type: "payment_required"` または `type: "payment_declined"` であり - 素の `500` ではありません。

## Forter による支援

エージェント決済の最も難しい部分は `402` のハンドシェイクではありません - ウォレットを巡るすべて、すなわちウォレットを作り、資金を入れて維持し、それを通じて取引する当事者の KYC / KYA 義務を満たすことです。[**Forter エージェンティック・オーケストレーション・スイート**](https://www.forter.com/blog/agentic-orchestration/?utm_source=github&utm_medium=referral&utm_campaign=agentic-readiness-guide&utm_content=m4-8-payment-protocols) はゲートウェイで x402 と MPP を話し、そのウォレットの負担を引き受けられます - あなたの有料ルートの背後にウォレットを立ち上げて維持し、Forter Identity Network から引いたアイデンティティと KYC/KYA のチェックを各清算に適用します。

オーケストレーションスイートのこの部分は初期段階の提供であり、プロトコルそのものとともに動きます - 今日プロダクション対応のものと、ロードマップ上のものを、Forter とともにスコープしてください。あなたのものとして残るもの: あなたの価格設定と、清算の背後にあるプロセッサとの関係。

# 4.9  エージェンティックコマースプロトコルに対応する

## 概要と理由
エージェントがユーザーに代わって買い物をするとき - 製品を見つけ、カートを組み立て、チェックアウトし、販売後には注文状況を確認し、返品を開始し、課金に異議を申し立てる - それは、あなたのストアフロント HTML のスクレイプではなく、構造化されたコマースのサーフェスを必要とします。これをカバーするプロトコルが2つあります。**UCP**（Universal Commerce Protocol）と **ACP**（Agentic Commerce Protocol）です。どちらも、エージェントがカタログを発見し、カートを組み立て、定められた契約に対して購入を完了できるようにします。このガイドラインは、エージェントがあなたの製品を閲覧し、ユーザーに代わってチェックアウトしうる場合に当てはまります。メーター制の従量課金アクセスのみを販売しているなら、それは代わりに [4.8](./m4-8-payment-protocols.md) で、エージェント呼び出しで何も販売しないなら、両方を飛ばしてください。

これらの標準はまだ落ち着いていませんが、採用は本物で加速しています - マーチャントは今日、Google、Gemini、AI モードのサーフェスを通じて取引しています。

## スコアリング
- **工数 4/5** - 本物のコマースのサーフェスです。カタログまたはフィードの登録、カートの状態、チェックアウトフロー、清算の配線 - そのすべてが、動き続ける仕様の上にあります。
- **インパクト 4/5** - 新興ですが変革的です。エージェントに製品を販売する誰にとっても、これは急速に、売上が生まれる場所になりつつあります。
- **視覚的変化: なし** - 発見とチェックアウトのエンドポイント。ユーザーに見えるページは変わりません。

## 手順
1. **UCP に対応する。** あなたのサービス、機能、エンドポイントを宣言する `/.well-known/ucp` ディスカバリーファイルを公開します。UCP は Google のショッピンググラフの上に構築されるので、あなたの製品も Google Merchant Center にカタログとして掲載され - そして最新に保たれ - なければなりません。次に、UCP 仕様に完全準拠して `/checkout-sessions` を実装し、それが定義するすべてのリクエストとレスポンスの形を満たして、エージェントがカートを組み立て、UCP が *payment handler* と呼ぶものを通じて購入を完了できるようにします。
2. **ACP に対応する。** `/.well-known/acp.json` ディスカバリーファイルを公開し、ACP 駆動のエージェントがあなたの商品を発見できるよう、製品カタログを OpenAI に登録します。次に、ACP 仕様に完全準拠して `/checkout_sessions` を実装し、それが定義するすべてのリクエストとレスポンスを満たして、エージェントがチェックアウトをエンドツーエンドで実行し、ACP が *delegated payment* と呼ぶものを通じて清算できるようにします。
3. **Webhook でプラットフォームを同期し続ける。** 完了したチェックアウトは、注文の生涯の始まりにすぎません。注文の完了、キャンセル、そしてすべての注文状況の変化（出荷、配達、返金）について、エージェントプラットフォームに Webhook を送り返して、エージェント - とそれが代わって動くユーザー - が、古いスナップショットではなく常に現在の状態を見られるようにします。
4. **ゲストを登録ユーザーに移す。** プロトコルのチェックアウトは、デフォルトでゲストチェックアウトです。アイデンティティのリンクおよび／または OAuth（[3.1](./m3-1-oauth-discovery.md)）を追加して、エージェントの購入が実在する再訪顧客のアカウントに紐づけられるようにします - すべてのエージェント売上を匿名のままにする代わりに、注文履歴、保存された設定、ロイヤルティを解放します。

## Forter による支援

エージェンティックコマースは、このモジュールのうち [**Forter エージェンティック・オーケストレーション・スイート**](https://www.forter.com/blog/agentic-orchestration/?utm_source=github&utm_medium=referral&utm_campaign=agentic-readiness-guide&utm_content=m4-9-commerce-protocols) が最も完全に肩代わりできる部分です - そして、スタックのどれだけを引き渡したいかに応じてスケールします。

- **製品フィード** - Forter は、UCP と ACP が動く元となるカタログフィードを構築・保守でき、あるいはあなたがすでに生成しているフィードから作業します。
- **トークン化と清算** - Forter は、両プロトコルの清算ステップのための PCI 準拠のトークナイザーであり、*かつ* あなたに代わって資金をオーソリ・キャプチャできる PSP 非依存の決済オーケストレータです - あるいは、あなたがすでに使っているトークン化とプロセッサと並んで組み込まれます。
- **カートとチェックアウト** - Forter は、あなたのカートとチェックアウトのフローを、既存の OMS や内部システムを回り込むのではなく、その *内側* で動かして運用します。
- **Webhook とアイデンティティ** - すでにオーケストレーションスイートを連携済みの Forter 顧客なら、手順3の注文 Webhook と状況更新はあなたのために処理され、Forter のアイデンティティリンクが組み込まれています - そのため、手順3と4はわずかな追加作業で着地します。
- **購入後** - 返品、返金、チャージバックの異議申し立ては、販売後に損失が集中する場所であり、エージェントはユーザーに代わってこの3つすべてを開始します。Forter の **Abuse Prevention** は、あなたが承認する前に返品・返金リクエストを悪用についてスコアリングします。**Dispute Management** は、着地した異議についてチャージバックの再表明 (representment) と回収を処理します - どちらも、チェックアウトをクリアしたのと同じ Forter Identity Network のシグナルを活用します。
- **仕様の変動** - UCP と ACP はまだ動いています。Forter はすべての改訂を追跡するので、仕様がその下で変わってもあなたの連携は壊れません。

より深い価値は、これらのプロトコルが *運ばない* ものにあります。現状では、UCP も ACP も、健全な検証・不正判定に必要なすべてのシグナルを表面化しません。Forter は、欠けている部分 - アイデンティティ、デバイスの評判、行動履歴、エージェントの来歴 - を収集し、カートと並べて1つの一貫した検証・不正チェック呼び出しに組み立てます。その対応付けと収集の作業はかなりのもので、過小評価されがちです。それを肩代わりすることが、単に完了するチェックアウトと、信頼できるチェックアウトの差です。オーケストレーションスイートが仲介するすべてのチェックアウトは、人間のカード非提示 (card-not-present) コマースを動かすのと同じ Forter Identity Network のシグナルを運びます。

あなたのものとして残るもの: あなたのカタログ、あなたの価格設定、そして、どれだけ持ち込み、どれだけ引き渡すかの判断。
